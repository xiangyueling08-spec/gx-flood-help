# 广西洪灾互助信息平台

> Until everyone is safe.

## 写在前面

这个项目的出发点很简单：当你的家人朋友身处受灾地区，而你远在他乡的时候，希望能有一个地方，让求助的人和能帮忙的人更容易被彼此看见——直到每一个人都平安。

受限于个人精力和传播渠道，这个版本大概率赶不上眼下这场广西洪灾，帮不上太多忙，这是目前最大的遗憾。但如果能有更多人愿意一起打磨它，下一次灾害发生时，或许我们可以在第一时间拿出一个更成熟的版本——让等待消息的人少一点无助，让每一个需要帮助的人，多一条能被看见的求助路径。

## 一、这个平台是做什么的

一个轻量的信息互助工具，解决的核心问题是：**灾区外的人如何有效地为灾区内的家人朋友求助，本地人看到线索之后又该往哪里反馈**。目前包含两个核心功能：

### 登记求助
任何人都可以为受灾/失联的家人朋友登记一条求助信息，字段包括：
- 求助人联系方式、与被寻人的关系、被寻人姓名或称呼
- 年龄段、最后联系时间、最后已知位置
- 是否行动不便、是否有基础病、是否需要紧急救援
- 详细住址（选填，**默认不公开**，数据库权限层面强制隐藏，不是仅前端隐藏）

### 提交线索
本地人如果看到、了解到某条求助对应的情况，可以直接在对应条目下提交线索（看到的时间、位置、情况描述、是否有影像资料、联系方式选填可匿名），不需要自己联系求助人。

### 状态跟踪
每条求助有六种状态：**待核实 / 已有线索 / 已转移安置 / 已联系上 / 需要紧急救援 / 已关闭**。有新线索提交时会自动从"待核实"推进到"已有线索"；其余状态变更需要管理员登录才能操作，避免状态被随意篡改。

### 基础防滥用
蜜罐字段拦截机器人、提交频率限制、诈骗关键词过滤、举报满一定次数自动隐藏待复核。**这些都只是初步防护**，还没有手机号验证，防不住存心搞破坏的人，见下文"后续可以做的事"。

### 技术栈
纯前端（HTML/CSS/JS，无需构建工具）+ Supabase（Postgres 数据库、行级权限 RLS、Auth 登录），部署在 Vercel。

## 二、投入正式使用前，需要认真考虑的问题

这一部分是诚实的风险清单，不是免责声明，是真的希望后来者认真读一下再决定要不要大规模推广。

- **法律与合规风险**：会收集姓名、联系方式、位置、住址等个人信息，落在《个人信息保护法》规制范围内；如果想稳定覆盖国内用户，还涉及 ICP 备案与内容审核责任的权衡（海外托管免备案但访问不稳定，国内备案稳定但流程慢、需接受监管）。
- **和已有平台的关系**：这次广西洪灾期间，微博已经开通"广西内涝求助"超话并配有小程序表单，官方会整理上报救援通道。这说明我们不是填补空白，而是提供一个可按状态/地区筛选的补充工具，推广时应坦诚告知用户"官方渠道优先"，避免信息进一步分散。但小红书上许多求助贴的存在，也说明一个信息整合平台是有意义的，所以我们做这件事是有意义的。
- **推广本身的难度**：没有本地志愿者组织、公益机构背书，工具再好也没人用；而灾害窗口期很短，推广速度往往跟不上灾情——这是需要提前想清楚的现实局限。

## 三、我们需要你的帮助

目前是一个非常初级的版本，比起再多写点代码，更缺的是这两类专业判断：

### 1）法律意见
- 收集、展示、留存姓名/联系方式/住址等个人信息，是否符合《个人信息保护法》的要求？需要补充哪些用户告知或授权机制？
- 现在页面上的"免责声明"是通用模板措辞，没有经过专业审核，条款实际能起到多少效力？

### 2）技术支持
- 真正的手机号验证（接入短信服务商），减少虚假信息和恶意刷屏
- 更强的反滥用机制：设备指纹、图形验证码、基于数据库的持久化限流
- 敏感词库从写死在代码里，改成后台可配置
- 多管理员协作：操作日志、权限分级
- 弱网/离线场景优化——灾区常常断网断电，如何让提交更容易成功
- 地图可视化，把求助信息叠加到地图上展示
- 和现有大平台（微博超话等）数据互通的可能性
